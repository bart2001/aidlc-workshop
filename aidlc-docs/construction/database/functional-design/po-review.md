## 2. 비즈니스 규칙 적합성 검증

### ✅ 적합한 규칙

| 규칙 | 비즈니스 적합성 |
| --- | --- |
| 주문 상태 전이 (PENDING→PREPARING→COMPLETED) | 실제 매장 운영 흐름과 일치 (접수→조리→완료) |
| 주문번호 매일 리셋 (001부터) | 매장 운영에 적합, 고객이 기억하기 쉬운 번호 |
| 테이블당 활성 세션 1개 제한 | 동시에 여러 고객이 같은 테이블 사용 방지 |
| 세션 종료 시 OrderHistory로 이동 | 과거 이력 보존하면서 현재 주문 깔끔하게 정리 |
| 메뉴 삭제 시 OrderItem.menu_item_id = NULL | 메뉴 삭제해도 기존 주문 이력 유지 (스냅샷) |
| 관리자 계정 잠금 (5회 실패, 30분) | US-A01 수용기준 5번 충족 |
| 멀티테넌트 격리 (store_id) | 매장 간 데이터 완전 분리 |

### ⚠️ 확인 필요 사항

| 항목 | 내용 | 영향도 | 판단 |
| --- | --- | --- | --- |
| 주문 삭제 시 총 주문액 재계산 | business-rules.md에 "삭제는 모든 상태에서 관리자만 가능"으로 명시됨. US-A03-2 수용기준 3번(총 주문액 재계산)은 백엔드 로직에서 처리 필요 | 낮음 | DB 설계 범위 밖, Backend에서 처리 예정 |
| OrderHistory 날짜 필터 | US-A03-4 수용기준 3번에 "날짜 필터" 요구. OrderHistory에 completed_at 인덱스 있음 | 낮음 | ✅ 인덱스로 지원 가능 |
